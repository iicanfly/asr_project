# ENV_MATRIX

## 总体说明
- 本项目最终部署目标是内网环境。
- 所有开发任务都必须按`双环境`思路设计和验证。
- 外网用于功能开发与调试，内网用于最终落地。
- 由于当前无法直接调试内网，任何实现都应避免破坏现有内网可用路径。

## 环境切换方式
- 使用 `.env` 中的 `USE_INTRANET` 开关区分环境。
- `USE_INTRANET=False`：外网开发模式。
- `USE_INTRANET=True`：内网部署模式。

## 服务与调用差异

| 项目 | 外网 | 内网 | 当前状态 |
| --- | --- | --- | --- |
| LLM 服务 | OpenAPI 接口 | OpenAPI 接口 | 已调通 |
| ASR 服务 | OpenAPI 格式 | HTTP `POST` 请求 | 已调通 |
| 鉴权方式 | 与内网一致 | 与外网一致 | 已调通 |
| 证书生成方式 | 与内网一致 | 与外网一致 | 已调通 |

## 关键结论
- LLM 在内外网的调用方式一致，改动时通常可复用同一套接口抽象。
- ASR 在内外网的调用方式不同，是当前最需要谨慎保护的环境差异点。
- 当前项目代码中的内外网调用链路已经验证通过，后续改造应优先保持兼容。

## 部署与依赖约束
- 外网调通后，通过 `conda pack` 打包虚拟环境。
- 打包产物迁移到内网后，通过解压方式使用。
- 新增依赖时，需要优先考虑其是否适合被 `conda pack` 打包与迁移。

## 开发注意事项
- 任何涉及 ASR 调用链路的改动，都需要明确检查外网 OpenAPI 路径和内网 HTTP `POST` 路径是否同时兼容。
- 不应因为外网调试方便，就默认内网路径可以自然成立。
- 任何“仅外网可跑通”的实现都不能视为任务完成。
