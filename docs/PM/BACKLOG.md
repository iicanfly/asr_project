# BACKLOG

## 维护原则
- 本项目所有开发任务默认都属于`双环境`任务。
- 外网用于开发与调试，内网是最终部署环境。
- 任何修改都必须同时考虑内网与外网行为是否一致可控。
- 由于当前无法直接在内网调试，所有实现需要优先保证内网路径的稳妥性，避免破坏已验证通过的内网逻辑。

## 当前优先级

### P0：实时转写质量提升
#### 1. 静音音频优化
- 目标：旁边人小声说话、低价值背景音、不应进入转写主流程。
- 原因：当前实时转写会误收环境中的微弱无效声音，影响质量和稳定性。
- 网络范围：双环境。
- 验收标准：明显的低音量背景声不再频繁触发转写输出。

#### 2. 语气词过滤
- 目标：过滤“嗯”“啊”“呀”等低信息量语气词。
- 原因：当前输出中语气词过多，影响文本质量。
- 网络范围：双环境。
- 验收标准：常见语气词显著减少，且不误伤正常短句。

#### 3. 多层次音频转写
- 目标：前端先显示小段实时结果，随后在更大音频片段形成后，用更高质量结果替换前面的粗糙结果。
- 原因：纯小段实时转写虽然快，但错误率高，需要通过大段复写提升整体可读性与准确率。
- 网络范围：双环境。
- 验收标准：前端能看到“先出结果、后被更高质量结果替换”的稳定行为。
- 当前状态：
  - 基础协议与最小闭环已开始落地，下一步重点变成“真实录音效果调优 + 分段边界配合验证”。

#### 4. 转写文本分段策略
- 目标：避免点击开始录音后所有转写内容一直堆在同一个段落中，同时避免分段过碎。
- 原因：当前没有分段机制，导致内容结构差；但过于频繁分段也会降低可读性。
- 网络范围：双环境。
- 验收标准：转写结果具备合理段落边界，能兼顾连续性与可读性。
- 当前状态：
  - 已开始从“纯时间窗合并”转向“后端 `segment_id` + 句子边界启发式 + 前端智能拼接”的组合策略。
  - 下一步重点变成真实录音下的边界调优，以及观察是否仍有过碎或过黏现象。

## 暂缓事项
- 音频上传功能当前可用，但暂不作为近期重点优化对象。
- 文档生成功能当前可用，但暂不作为近期重点优化对象。

## 当前阶段目标
- 在不破坏现有内外网调用链路的前提下，优先大幅提高实时转写质量。

## 2026-05-10 进度补充
- 长时间录音稳定性已经开始进入第一轮代码级防护：
  - 优先削减前端长时间录音时不必要的内存占用与缓存写放大。
- 停止录音收尾链路已经开始补强：
  - 优先减少“停止太快导致最后一小段尾音丢失”的问题。
- 本地缓存容量保护已经开始补强：
  - 优先降低缓存超限或缓存损坏后导致前端恢复异常的风险。
- 双层转写结果去重已经开始补强：
  - 优先减少无意义的重复 partial / rewrite 回推。
- 前端结果投递去重已经开始补强：
  - 优先减少同一 `result_id` 重复到达时的重复渲染。
- 前端消息 ID 一致性已经开始补强：
  - 优先减少缓存恢复后新旧消息 ID 碰撞的风险。
- 静音过滤 / 弱语音优化已进入第二轮规则细化：
  - 不再让 `active_ratio` 单独放行实时切片，而是增加“最小有声时长 / 最小活跃时长 / 柔和语音兜底”的组合门槛。
  - 下一步重点变成真实录音校准：确认误抑制旁边小声说话的同时，没有把主说话人的轻声内容一起杀掉。
  - 尾静音路径也已开始单独收紧：短促弱语音在尾静音到来时将更容易被直接丢弃，而不是被切成一条独立转写。
