# KNOWN_ISSUES

## 说明
- 本文件记录当前已经确认存在或高度疑似存在的问题、限制和结构性风险。
- 这里只记录从现有代码、配置或当前协作结论中能明确看出的内容。

## P0：当前核心产品问题

### 1. 实时转写输出过碎
- 现象：
  - 经常几个字就形成一条结果。
  - 完整句子容易被切断。
- 当前代码依据：
  - 后端按时长阈值和静音检测触发转写。
  - 前端只做“同说话人 + 10 秒内”的简单合并。
- 影响：
  - 可读性差。
  - 后续文档生成输入质量也会下降。

### 2. 弱背景语音会误进入转写
- 现象：
  - 旁边人小声说话也可能被识别出来。
- 当前代码依据：
  - 现有 `detect_silence()` 规则较简单，主要依赖 RMS、最大幅值、静音帧比例。
- 影响：
  - 噪声混入主转写内容。
  - 实时质量下降。

### 3. 语气词过滤能力不足
- 现象：
  - “嗯”“啊”“呀”等低信息量内容偏多。
- 当前代码依据：
  - `should_filter_asr_result()` 确实存在过滤逻辑，但规则较粗。
  - 当前过滤字符串本身还存在可疑编码异常。
- 影响：
  - 转写文本冗余，阅读体验差。

### 4. 缺少“小段结果 -> 大段结果”的替换回写机制
- 现象：
  - 当前只能先识别一段、展示一段，没有后续高质量结果回写前文的机制。
- 当前代码依据：
  - 后端回推的 `asr_result` 当前直接作为最终展示结果。
  - 前端 `handleASRResult()` 只有追加和合并逻辑，没有替换已有片段的能力。
- 影响：
  - 实时快，但准确率和最终可读性受限。

### 5. 还没有真正的段落级分段策略
- 现象：
  - 不希望所有内容都堆成一个段落，也不希望段落被切得太碎。
- 当前代码依据：
  - 前端当前只有消息级合并，没有更高层的段落语义。
- 影响：
  - 用户阅读和后续整理成本较高。

## P0：当前链路级实现风险

### 6. 停止录音时没有显式通知后端结束 session
- 现象：
  - 后端定义了 `@socketio.on("stop_recording")`
  - 但前端代码中没有发现对应的 `socket.emit('stop_recording', ...)`
- 影响：
  - 用户点击“停止录音”后，后端 session 可能仍保留。
  - 临时文件句柄与内存缓冲只会在 socket 断开时清理。
  - 这可能影响下一次继续录音时的状态边界。

### 7. Electron 等待的后端端口与 Flask 默认端口不一致
- 现象：
  - Electron 主进程 `src/main/main.js` 中等待端口是 `8000`
  - Flask 默认端口来自 `.env`，当前为 `6543`
- 影响：
  - Electron 自动拉起链路可能并不完全匹配当前 Web 后端配置。
  - 后续桌面端联调与打包前需要专门核对。

### 8. Web 与 Electron 有两套前端入口，存在漂移风险
- 现象：
  - Web 入口：`templates/index.html + static/*`
  - Electron 入口：根目录 `index.html`
- 影响：
  - 改一个入口后，另一个入口可能不同步。
  - 前端功能修复和 UI 调整容易出现“浏览器可用、桌面端没同步”的问题。

## P1：配置与运维风险

### 9. `.env` 当前存在被纳入仓库管理的风险
- 现象：
  - `.gitignore` 只忽略 `.env.*`
  - 没有忽略根级 `.env`
  - 当前仓库中存在 `.env`
- 影响：
  - 如果 `.env` 中包含真实服务地址、密钥或内网信息，存在泄露和环境污染风险。
- 备注：
  - 这条是配置管理风险，不影响当前功能理解，但后续应谨慎处理。

### 10. 新增依赖必须考虑 `conda pack` 迁移约束
- 现象：
  - 当前部署方式依赖外网打包、内网解压使用。
- 影响：
  - 新依赖若不适配该流程，可能外网能开发、内网无法落地。

## P1：代码与文本质量风险

### 11. 部分源码中的中文字符串存在编码异常迹象
- 现象：
  - 多个源码文件中的中文字符串在当前终端视角下表现为乱码。
  - 包括提示词、UI 文本、过滤词等区域。
- 影响：
  - 可能影响后续维护判断。
  - 如果不是单纯的终端显示问题，而是历史编码已损坏，可能影响实际运行文本质量。
- 备注：
  - 当前至少可以确认“终端显示编码”存在问题。
  - 是否存在“文件内容本身已发生编码污染”，后续需要按文件逐步确认。

## 当前建议的处理顺序
1. 先处理实时转写质量主问题：
   - 静音过滤
   - 语气词过滤
   - 双层转写与回写替换
   - 分段策略
2. 尽快核对停止录音的后端 session 清理问题。
3. 在桌面端继续推进前，核对 Electron 端口与入口结构问题。
4. 在后续某一轮中单独处理配置安全与编码一致性问题。

## 2026-05-10 新增已知状态修正
- 已修复：
  - 前端停止录音时未显式通知后端结束 session。
  - 前端实时消息在“UI 已合并、数据仍重复 push”场景下的数据不一致问题。
- 新的剩余风险：
  - `stop_recording` 当前只负责结束 session 与清理缓冲，不会强制把不足常规切片阈值的尾部短音频送去转写。
  - 因此，如果用户在一句话刚结束或尚未形成完整切片时立即停止录音，最后一小段尾音仍可能丢失。
- 已降级为“待启用能力”而非“零准备”：
  - 前端和后端 payload 已预留 `result_id / segment_id / replace_target_id`，但真正的“大段结果回写替换小段结果”仍未实现。
- 当前建议的处理顺序补充：
  1. 先继续做静音过滤、弱语音、语气词过滤。
  2. 然后评估是否需要引入 `stop flush` 策略处理尾部短缓冲。
  3. 再推进双层转写与替换回写。
