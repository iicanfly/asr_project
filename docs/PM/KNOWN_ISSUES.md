# KNOWN_ISSUES

## 说明
- 本文件记录当前已经确认存在或高度疑似存在的问题、限制和结构性风险。
- 这里只记录从现有代码、配置或当前协作结论中能明确看出的内容。

## P0：当前核心产品问题

### 1. 实时转写输出过碎
- 现象：
  - 经常几个字就形成一条结果。
  - 完整句子容易被切断。
- 当前代码依据：
  - 当前已加入 `segment_id` 驱动的段内合并、段级回写，以及“句子边界 / 足够长文本”辅助收尾。
  - 前端同段文本也已改为智能拼接，不再对中文一律插空格。
- 影响：
  - 已明显缓解，但边界阈值仍需真实录音继续调优。
  - 如果边界判断偏紧，仍可能过碎；如果偏松，也可能出现段落过长。

### 2. 弱背景语音会误进入转写
- 现象：
  - 旁边人小声说话也可能被识别出来。
- 当前代码依据：
  - 现有 `detect_silence()` 规则较简单，主要依赖 RMS、最大幅值、静音帧比例。
- 影响：
  - 噪声混入主转写内容。
  - 实时质量下降。

### 3. 语气词过滤能力不足
- 现象：
  - “嗯”“啊”“呀”等低信息量内容偏多。
- 当前代码依据：
  - 当前已经加入“边界语气词清洗 + 低信息量短句过滤”，比最初粗规则更细。
  - 但这类规则仍主要是启发式判断，效果仍需要真实录音继续校准。
- 影响：
  - 如果规则偏松，语气词仍会漏出；如果规则偏严，短句可能被误删。

### 4. 缺少“小段结果 -> 大段结果”的替换回写机制
- 现象：
  - 当前只能先识别一段、展示一段，没有后续高质量结果回写前文的机制。
- 当前代码依据：
  - 当前已经具备最小闭环：后端会在段内累计多块音频，并在条件满足时再次识别更长上下文，然后通过 `replace_target_id` 回写替换。
  - 前端已经支持按 `segment_id` 继续合并或切换新段。
- 影响：
  - 基础能力已不再是“完全缺失”，但阈值、时机和真实效果仍需调优。

### 5. 还没有真正的段落级分段策略
- 现象：
  - 不希望所有内容都堆成一个段落，也不希望段落被切得太碎。
- 当前代码依据：
  - 当前已新增 `segment_id` 驱动的段内合并 / 跨段拆分，并加入“句子边界 / 足够长文本”辅助收尾。
  - 但段落边界仍主要是启发式规则，不是真正的语义级分段模型。
- 影响：
  - 用户阅读和后续整理成本较高。

## P0：当前链路级实现风险

### 6. 停止录音时没有显式通知后端结束 session
- 现象：
  - 后端定义了 `@socketio.on("stop_recording")`
  - 但前端代码中没有发现对应的 `socket.emit('stop_recording', ...)`
- 影响：
  - 用户点击“停止录音”后，后端 session 可能仍保留。
  - 临时文件句柄与内存缓冲只会在 socket 断开时清理。
  - 这可能影响下一次继续录音时的状态边界。

### 7. Electron 等待的后端端口与 Flask 默认端口不一致
- 现象：
  - Electron 主进程 `src/main/main.js` 中等待端口是 `8000`
  - Flask 默认端口来自 `.env`，当前为 `6543`
- 影响：
  - Electron 自动拉起链路可能并不完全匹配当前 Web 后端配置。
  - 后续桌面端联调与打包前需要专门核对。

### 8. Web 与 Electron 有两套前端入口，存在漂移风险
- 现象：
  - Web 入口：`templates/index.html + static/*`
  - Electron 入口：根目录 `index.html`
- 影响：
  - 改一个入口后，另一个入口可能不同步。
  - 前端功能修复和 UI 调整容易出现“浏览器可用、桌面端没同步”的问题。

## P1：配置与运维风险

### 9. `.env` 当前存在被纳入仓库管理的风险
- 现象：
  - `.gitignore` 只忽略 `.env.*`
  - 没有忽略根级 `.env`
  - 当前仓库中存在 `.env`
- 影响：
  - 如果 `.env` 中包含真实服务地址、密钥或内网信息，存在泄露和环境污染风险。
- 备注：
  - 这条是配置管理风险，不影响当前功能理解，但后续应谨慎处理。

### 10. 新增依赖必须考虑 `conda pack` 迁移约束
- 现象：
  - 当前部署方式依赖外网打包、内网解压使用。
- 影响：
  - 新依赖若不适配该流程，可能外网能开发、内网无法落地。

## P1：代码与文本质量风险

### 11. 部分源码中的中文字符串存在编码异常迹象
- 现象：
  - 多个源码文件中的中文字符串在当前终端视角下表现为乱码。
  - 包括提示词、UI 文本、过滤词等区域。
- 影响：
  - 可能影响后续维护判断。
  - 如果不是单纯的终端显示问题，而是历史编码已损坏，可能影响实际运行文本质量。
- 备注：
  - 当前至少可以确认“终端显示编码”存在问题。
  - 是否存在“文件内容本身已发生编码污染”，后续需要按文件逐步确认。

## 当前建议的处理顺序
1. 先处理实时转写质量主问题：
   - 静音过滤
   - 语气词过滤
   - 双层转写与回写替换
   - 分段策略
2. 尽快核对停止录音的后端 session 清理问题。
3. 在桌面端继续推进前，核对 Electron 端口与入口结构问题。
4. 在后续某一轮中单独处理配置安全与编码一致性问题。

## 2026-05-10 新增已知状态修正
- 已修复：
  - 前端停止录音时未显式通知后端结束 session。
  - 前端实时消息在“UI 已合并、数据仍重复 push”场景下的数据不一致问题。
- 新的剩余风险：
  - `stop_recording` 已开始支持尾段 flush，但阈值仍是启发式规则。
  - 因此，如果最后一小段极短、极弱，或刚好落在阈值边界附近，仍可能出现“该补没补”或“补了但质量一般”的情况。
- 已降级为“待启用能力”而非“零准备”：
  - 前端和后端 payload 已预留 `result_id / segment_id / replace_target_id`，并且最小替换回写闭环已经落地；当前主要问题变成阈值与实际效果调优。
- 当前建议的处理顺序补充：
  1. 先继续做静音过滤、弱语音、语气词过滤。
  2. 然后评估是否需要引入 `stop flush` 策略处理尾部短缓冲。
  3. 再推进双层转写与替换回写。

## 2026-05-10 / 静音过滤阶段补充
- 已缓解但仍待观察：
  - 实时切片现在已经会识别“弱背景音 / 低可用语音密度”并直接丢弃缓冲，因此该问题已从“完全无防护”降为“阈值待调优”。
- 当前新增风险点：
  - 如果阈值设置过严，极弱但真实有效的人声可能被当作弱背景音丢弃。
  - 需要通过明早的真实录音回归，观察是否出现“降噪变强，但漏识别也变多”的副作用。

## 2026-05-10 / 语气词过滤阶段补充
- 已缓解但仍待观察：
  - 实时结果现在会先清洗两端语气词，再决定是否展示，因此“嗯”“啊”“呀”等纯低信息量输出预计会明显减少。
- 当前新增风险点：
  - 语气词清洗目前主要针对结果首尾，不会深入重写句中内容，因此长句中的口头禅仍可能残留。
  - 已主动移除默认过滤词中的“那”，以避免误伤“那我们开始”“那个方案”这类正常表达；是否还需要更强的句中口头禅处理，需结合明早录音效果再决定。

## 2026-05-10 / 替换回写阶段补充
- 已缓解但仍待观察：
  - 系统已经不再只有“小段直接当最终结果”这一种模式；满足条件后，后端会触发段级重识别并覆盖前面的粗结果。
- 当前新增风险点：
  - 当前段级回写会额外增加一次 ASR 请求，可能带来一些延迟和服务压力，需要通过明早的连续录音观察实际成本。
  - 段落边界目前仍主要依赖尾部静音和累计时长，如果真实说话节奏与阈值不匹配，仍可能出现“回写太勤”或“分段太碎 / 太黏”的问题。

## 2026-05-10 / 分段策略阶段补充
- 已缓解但仍待观察：
  - 段落结束现在不再只看尾部静音，还会结合当前文本是否像一个可收尾句子，能减少短静音导致的过早切段。
  - 前端同段中文文本合并时不再默认插入空格，整体阅读感更自然。
- 当前新增风险点：
  - “句子边界”仍是启发式判断，遇到口语化、方言化或无标点输出时，仍可能出现误判。
  - 英文 / 中文混合场景目前只做了基础拼接规则，复杂中英混排的展示效果仍需观察。

## 2026-05-10 / 长时间录音稳定性阶段补充
- 已缓解但仍待观察：
  - 前端现在只在后端离线 / socket 断开时才额外缓存原始音频，在线长录音时不会再把整段音频无限堆在 `audioChunks` 中。
  - 转写缓存写入已改成防抖模式，降低了长会话里频繁全量写 localStorage 的压力。
  - 本地缓存写入与读取现在已有统一保护，缓存损坏或超限时会降级成警告，而不是直接把页面逻辑打断。
- 当前新增风险点：
  - 如果后端长时间离线，离线补偿音频仍会继续累积，极端场景下仍可能占用较多前端内存。
  - localStorage 本身仍有浏览器容量上限，超长转写会话需要继续观察缓存体积、告警体验与恢复行为。

## 2026-05-10 / 停止录音收尾阶段补充
- 已缓解但仍待观察：
  - 停止录音时，后端现在会优先尝试处理剩余短尾音频，并在必要时补做一次最终段级回写，减少最后一句丢尾。
- 当前新增风险点：
  - stop flush 采用更宽松阈值后，理论上也更容易把某些边界噪声误当成尾段语音，需要结合真实录音继续校准。
  - 如果用户在非常高频的连续点击 / 极端网络延迟场景下停止录音，仍需观察“延迟清理”路径是否足够稳。

## 2026-05-10 / 结果去重阶段补充
- 已缓解但仍待观察：
  - 对与当前展示文本实质相同的 partial / rewrite，系统现在会直接跳过，不再反复刷新前端或重复写缓存。
  - 前端对同一个 `result_id` 的重复投递也开始直接跳过，进一步减少网络边界造成的重复处理。
- 当前新增风险点：
  - 去重当前基于“清洗后紧凑文本是否变化”的启发式判断；如果某些细微但有价值的改写只体现在标点或极少数字词差异上，仍需观察是否会被过度抑制。

## 2026-05-10 / 前端消息 ID 一致性补充
- 已缓解但仍待观察：
  - 缓存恢复后的消息 ID 现在会同步推进 `messageIdCounter`，降低新增消息与旧消息撞 ID 的风险。
- 当前新增风险点：
  - 这类一致性问题主要仍依赖前端单机状态维护；后续如果引入更复杂的多来源消息流，仍需继续观察是否需要更强的 ID 生成策略。
## 2026-05-10 / localStorage 读取保护阶段补充
- 已缓解但仍待观察：
  - localStorage 相关路径现在不再只保护写入，还补上了读取异常保护；浏览器限制本地存储访问时，前端会优先降级告警。
- 当前新增风险点：
  - 当前告警仍集中在页面内提示与控制台日志，是否足够让普通用户理解“缓存为什么没恢复”，仍需实际体验验证。
  - 如果未来引入更多新的本地缓存键，需要继续遵守安全读取封装，否则可能重新出现漏网读取路径。

## 2026-05-10 / 静音过滤第二轮补充
- 已缓解但仍待观察：
  - “有活跃度但几乎没有有声支撑”的缓冲，现在不会再只凭 `active_ratio` 被视为可转写语音。
  - 系统开始同时关注活跃 / 有声持续时长，能更好地区分“持续柔和主语音”和“零散弱背景音”。
- 当前新增风险点：
  - 当前时长门槛属于启发式规则，若主说话人音量极低且讲话非常断续，仍可能出现边界漏识别。
  - 新的兜底路径依赖 RMS、峰值和少量有声占比的组合，真实会议环境下是否足够稳，还需要继续看录音日志与前端文本效果。

## 2026-05-10 / 尾静音短促弱语音补充
- 已缓解但仍待观察：
  - 尾静音触发路径现在会额外过滤“只有一点短促弱语音、随后立刻静音”的片段，减少旁边轻声插话被单独切出的概率。
- 当前新增风险点：
  - 对极短主语音（如非常短的“好”“嗯行”）来说，新的尾段门槛可能仍有误杀风险，需要结合真实录音判断是否值得保留这类超短词。

## 2026-05-10 / voiced density 补充
- 已缓解但仍待观察：
  - 柔和语音兜底现在还会看 `voiced_density`，因此“活跃很多但有声成分很稀”的片段更不容易被误放行。
- 当前新增风险点：
  - 如果主说话人本身断续、咬字很散，新增加的 density 门槛仍可能带来边界漏识别，需要结合真实录音继续校准。

## 2026-05-10 / 样本覆盖度补充
- 已确认但仍待解决：
  - 当前仓库旁边现有的 `41.wav / 70.wav / 97.wav` 离线回放结果都落在 `strong_signal`，更像主语音样本。
- 当前新增风险点：
  - 这意味着我们虽然已经连续细化了弱背景过滤规则，但仍缺少足够真实的“旁边人小声插话 / 低音量背景说话”样本来验证规则是否真的命中目标场景。
