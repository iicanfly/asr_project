# SESSION_SUMMARY

## 当前项目状态
- 实时录音功能已可用，但实现仍较初步。
- 音频上传功能已可用，但当前不作为近期优化重点。
- 文档生成功能已可用，但当前不作为近期优化重点。
- 项目管理文档与项目认知文档体系已建立，可用于后续会话快速恢复上下文。

## 当前核心目标
- 在不破坏现有内外网调用链路的前提下，大幅提升实时转写质量。

## 当前最关注的功能范围
- 当前阶段聚焦实时录音转写。
- 音频上传与文档生成先维持现状，暂不优先调整。

## 实时转写当前已知问题
- 输出过碎，经常只有几个字就形成一句，完整表达被切断。
- 旁边人说话的小声音也会被误转写。
- 还没有分段功能，但也不希望段落被切得过碎。
- 语气词过多，影响文本质量。
- 还没有“小段结果到大段结果的替换回写”机制。

## 当前临时决策
- 音频上传功能先按现状保留，不作为近期重点改造对象。
- 文档生成功能先按现状保留，不作为近期重点改造对象。
- 所有新开发与调优工作优先服务于实时转写质量提升。
- Codex 后续默认同步维护 `docs/PM/` 下的项目管理文档，并将文档更新纳入本地版本管理。

## 环境结论
- 本项目所有任务都按双环境处理。
- 外网用于调试，内网用于最终部署。
- 内网路径必须被特别谨慎地保护，因为当前无法直接在内网调试。

## 后续建议
- 优先梳理实时转写主链路中的音频切片、静音判定、前端展示与结果替换机制。
- 在开始较大改造前，补充一批项目认知文档，降低后续会话重复扫描代码的成本。

## 2026-05-10 / Sprint 0 状态更新
- 已完成正式开发前的小型稳态整理，目标是先稳住实时转写主链路，再进入质量优化。

  - 前端 `handleASRResult()` 已兼容 `result_id / segment_id / replace_target_id`，为后续“小段结果 -> 大段结果替换回写”预留数据结构。
  - 修复了“UI 已合并、但 `transcriptData` 仍重复追加”的一致性问题，避免后续文档生成和缓存数据与界面展示不一致。

- 当前仍未解决的核心产品问题没有变化：
  - 静音过滤与弱语音平衡仍需专项优化。
  - 语气词过滤规则仍较粗。
  - 还没有真正启用“小段结果 -> 大段结果”的回写替换策略。
  - 还没有段落级分段策略。
- 下一步建议：
  1. 先做静音过滤 / 弱语音识别的规则升级。
  2. 再做语气词过滤精细化。
  3. 最后进入双层转写与前端替换回写。

## 2026-05-10 / 实时转写第二轮进展
- 已继续处理第二优先级事项：语气词过滤精细化。
- 当前实现状态：
  - 后端新增 `refine_asr_result_text()`，会在发送前清洗实时结果两端的语气词和多余标点。
  - `should_filter_asr_result()` 已从粗粒度比例判断，升级为“边界清洗 + 低信息量短句过滤 + 有效短句白名单”组合规则。
  - `main.py` 发送实时结果时，默认优先使用清洗后的文本，并记录原始文本与清洗后文本，方便后续对照。
  - 已主动移除默认过滤词中的“那”，降低误伤正常中文短句的风险。
- 当前效果判断：
  - 纯“嗯”“啊”“哈”这类结果会直接被拦截。
  - “嗯 今天开始”“啊好的”“你好啊”这类结果会先收敛成更干净的文本再展示。
- 当前还未完成的点：
  - 句中口头禅仍未做更激进处理，避免当前阶段过早误伤正常内容。
  - 双层转写替换回写、分段策略、长时间录音稳定性回归仍在后续待办中。

## 2026-05-10 / 实时转写第三轮进展
- 已开始处理第三优先级事项：小段结果到大段结果的替换回写。
- 当前实现状态：
  - 后端会为每个实时会话维护 `active_segment`，把多个小 chunk 的音频累计到一个段内。
  - 当段内累计时长 / chunk 数达到阈值，或遇到较明显的尾部静音时，后端会对更长的段音频再做一次 ASR，并发出 `segment_rewrite` 结果。
  - 前端已经根据 `replace_target_id` 覆盖旧文本，并根据 `segment_id` 决定是否继续并入同一段落。
  - 这意味着当前链路已经从“单层小 chunk 输出”升级成“先快出、后回写”的基础双层模型。
- 当前效果判断：
  - 实时速度仍由小 chunk 保底。
  - 更长上下文的重识别现在有机会把同一段内前面的粗糙文本替换成更完整的表达。
- 当前还未完成的点：
  - 段级回写阈值还需要结合真实录音继续调优。
  - 分段策略虽然已有 `segment_id` 基础，但仍未达到最终理想状态。
  - 长时间录音稳定性回归尚未开始系统验证。

## 2026-05-10 / 实时转写第四轮进展
- 已继续处理第四优先级事项：更合理的分段策略。
- 当前实现状态：
  - 后端段落结束不再只靠“尾部静音 + 时长”，而是新增“句子边界 / 足够长文本”辅助判断。
  - `active_segment` 会记录当前段最近一次可展示文本，供分段收尾逻辑参考。
  - 前端同段结果合并时，已经改为智能拼接：中文片段默认直接相连，英文 / 数字连续词之间才保留空格。
- 当前效果判断：
  - 这轮的目标不是增加更多段，而是减少“不该断时断开”和“明明已合并但看起来仍很碎”的问题。
  - 预计同一段的中文显示会更自然，短静音导致的提前切段会有所减少。
- 当前还未完成的点：
  - 句子边界启发式仍需要真实口语录音继续校准。
  - 长时间录音稳定性回归尚未开始系统验证。

## 2026-05-10 / 实时转写第五轮进展
- 已开始处理第五优先级事项：长时间录音稳定性。
- 当前实现状态：
  - 前端录音过程中，原始音频片段只在后端离线 / socket 断开时才进入本地补偿缓存；在线正常转写时不再无限累积。
  - 转写缓存写入已改为防抖批量写，并在停止录音、页面卸载前强制落盘一次。
  - 离线补偿音频成功写入本地存储后，会立即清空本轮内存缓存，避免残留。
- 当前效果判断：
  - 这轮主要先处理最容易在长录音里放大的前端内存与缓存写放大问题。
  - 预计长时间录音时浏览器卡顿和内存上涨速度会有所缓解。
- 当前还未完成的点：
  - 仍需真实长录音回归，观察 localStorage 体积、浏览器内存、回写延迟与停止录音后的恢复行为。
