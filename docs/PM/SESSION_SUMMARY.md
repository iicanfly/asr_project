## 2026-05-11 / 当前实时转写状态补充
- 当前前端展示策略已经调整为：
  - 同一次连续录音过程中，默认先持续更新当前段。
  - 当一次 `high_rewrite` 完成后，如果当前稳定文本已达到约 100 字并且像完整句子，后端会在这里结束当前段。
  - 后续再次出现新的有效说话时，会创建新的 `segment_id`，前端随之新开下一条消息。
- 当前回写策略已经收敛为三级结构：
  - `partial`：小步实时结果，前端显示为深橙色。
  - `medium rewrite`：约 6 秒级滚动中级回写，前端显示为青绿色。
  - `high rewrite`：约 30 秒级高级回写，前端显示为黑色稳定文本。
- 当前新增收尾机制：
  - `IDLE_SEGMENT_SPLIT_SECONDS` 已默认关闭，不再把噪声环境下的活动判断作为主分段依据。
  - 当前主分段策略改为：`high_rewrite` 完成后，若稳定文本约满 100 字且像完整句子，就在该处开启新的段边界。
  - `IDLE_HIGH_REWRITE_SECONDS=10` 仍保留为长空闲场景下的兼容收尾路径，但不再负责主分段。
- 本轮重点不再是“分段策略”，而是：
  1. 保证高回写后的文本边界分段不会把一句完整话切碎；
  2. 保证三级回写在新分段策略下仍稳定触发；
  3. 继续观察延迟、漏字、语气词残留与长录音稳定性。
- 针对这轮联调过程中形成的调试教训，已新增长期记忆文档：
  - `docs/PM/REALTIME_DEBUG_PLAYBOOK.md`
  - 后续只要继续做实时转写相关开发，默认应先回看这份手册，再开始改代码。

## 实时转写当前已知问题
- 输出过碎，经常只有几个字就形成一句，完整表达被切断。
- 旁边人说话的小声音也会被误转写。
- 还没有分段功能，但也不希望段落被切得过碎。
- 语气词过多，影响文本质量。
- 还没有“小段结果到大段结果的替换回写”机制。

## 当前临时决策
- 音频上传功能先按现状保留，不作为近期重点改造对象。
- 文档生成功能先按现状保留，不作为近期重点改造对象。
- 所有新开发与调优工作优先服务于实时转写质量提升。
- Codex 后续默认同步维护 `docs/PM/` 下的项目管理文档，并将文档更新纳入本地版本管理。

## 环境结论
- 本项目所有任务都按双环境处理。
- 外网用于调试，内网用于最终部署。
- 内网路径必须被特别谨慎地保护，因为当前无法直接在内网调试。

## 后续建议
- 优先梳理实时转写主链路中的音频切片、静音判定、前端展示与结果替换机制。
- 在开始较大改造前，补充一批项目认知文档，降低后续会话重复扫描代码的成本。

## 2026-05-10 / 静音幻觉短句拦截补充
- 用户在真实口测中反馈：
  - 完全不说话时，前端仍会出现大量“嗯 / Yeah / Thank you / 你好 / What's that”之类的低信息混合短句。
- 本轮新增处理：
  - 在 `services/asr_service.py` 中补了“低信息片段”识别层。
  - 对整句按片段拆分后，会优先移除 `嗯 / 啊 / yeah / yes / huh / thank you / 你好 / 谢谢 / what's that` 等低信息片段。
  - 如果整句几乎全是这类低信息片段，则直接过滤，不再发往前端。
  - 如果句子里同时混有真实有效内容，则只剔除低信息片段，保留有效正文。
- 当前预期效果：
  - 静音阶段的幻觉短句应进一步下降。
  - 混合句中真正有信息量的中文内容应尽量保留。
- 当前剩余风险：
  - 真实用户如果单独说“你好 / 谢谢”等礼貌短句，现在更可能被过滤，需要结合真实使用场景继续评估是否接受。

## 2026-05-10 / 保守上传门控补充
- 用户进一步明确要求：
  - 静音片段不要上传，但又不能因为门槛过严而丢掉很短却可能重要的有声内容。
  - 当前实时链路的修改还要兼容后续多层次音频回写替换。
- 本轮新增处理：
  - 在 `services/asr_service.py` 中补了“疑似短语音先保留、不立刻上传”的门控分支。
  - 对 chunk 边界或尾静音边界处“不够确认但可能包含短语音”的片段，会先保留一小段缓冲，等待后续上下文再决定是否上传。
  - `stop flush` 也增加了“短促但连续的有声尾巴”放行分支，避免停止录音时把短语音直接吞掉。
- 兼容性原则：
  - 本轮没有把文本过滤结果直接反向写入段级音频累计逻辑。
  - 优先保证后续大段 rewrite 仍可以基于原始有效音频做回写，而不是因为这轮静音抑制把可用音频提前删掉。

## 2026-05-10 / 上下文型低信息尾巴清洗补充
- 用户最新真实反馈：
  - 说完一句真实内容后，虽然纯静音阶段的“嗯”减少了，但仍会看到 `Just / Okay / 对 / 是的吧 / 那啥 / Thank you / 嗯` 这类混合尾巴继续出现在前端。
- 本轮新增处理：
  - 在 `services/asr_service.py` 中补了“上下文型低信息片段”集合。
  - 像 `对 / 好的 / 是的 / 是的吧 / 好 / 那啥 / 那个 / okay` 这类短片段，不会被全局一刀切过滤；
  - 只有当它们混在一串明显低信息结果里时，才会与 `嗯 / thank you / just / what` 一起被清掉。
- 当前预期效果：
  - 保留单独出现的正常短句能力。
  - 继续压制“真实内容后面挂一串噪声尾巴”的前端展示。

## 2026-05-10 / 长尾巴截断补充
- 用户最新真实反馈：
  - 当前输出里，正文已经基本出来了，但结尾仍可能继续挂一长串“嗯。嗯。嗯。嘘。嗯。”。
- 本轮新增处理：
  - 在 `services/asr_service.py` 中补了“正文后的长尾巴截断”逻辑。
  - 如果一句有效正文后，尾部连续跟着多个低信息片段，系统会把整段尾巴直接裁掉，而不是继续保留某个可疑单字。
  - 同时把 `嘘` 纳入低信息片段集合，并把 `那 / 你好` 纳入多片段场景下的上下文型低信息短片段集合。
- 当前预期效果：
  - 比起前一轮，更适合解决“前半句基本正常，但后半段被嗯嗯嗯拖脏”的场景。

## 2026-05-10 / 实时转写第三轮进展
- 已开始处理第三优先级事项：小段结果到大段结果的替换回写。
- 当前实现状态：
  - 后端会为每个实时会话维护 `active_segment`，把多个小 chunk 的音频累计到一个段内。
  - 当段内累计时长 / chunk 数达到阈值，或遇到较明显的尾部静音时，后端会对更长的段音频再做一次 ASR，并发出 `segment_rewrite` 结果。

- 当前效果判断：
  - 实时速度仍由小 chunk 保底。
  - 更长上下文的重识别现在有机会把同一段内前面的粗糙文本替换成更完整的表达。
