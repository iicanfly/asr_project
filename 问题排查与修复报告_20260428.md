## 问题概述

实时录音功能无法正常输出语音识别文字，具体表现为：
1. 前端录音正常启动，麦克风权限获取成功
2. 后端日志显示 `Audio too long (30.0s), forcing processing`，但无识别结果返回

## 问题一：后端音频处理逻辑致命BUG（导致无识别输出）

### 现象

- 后端日志输出 `Audio too long (30.0s), forcing processing` 后，无任何ASR识别结果
- 前端收不到 `asr_result` 事件
- 整个会话永久卡死，后续所有音频数据被丢弃

### 根本原因

`on_audio_stream` 函数中，当触发条件满足时，代码设置 `session['processing'] = True` 后**直接 return**，导致：

1. **ASR处理代码永远不会执行**：处理代码位于函数下方，触发条件满足后直接退出函数，永远不会走到 ASR API 调用逻辑
2. **`processing` 标志永远不会重置**：`finally` 块中的 `session['processing'] = False` 不会被执行，导致会话永久卡在 processing 状态
3. **所有后续音频被丢弃**：函数开头 `if session['processing']: return` 会跳过所有后续音频包

**原代码逻辑流程（错误）**：
```
收到音频 → 检测到超30秒 → processing=True → return（退出函数）
                                        ↓
                              ASR处理代码（永远不会执行）
                              finally: processing=False（永远不会执行）
```

### 解决方案

引入 `should_process` 标志变量，将"判断是否需要处理"与"执行处理"分离：

1. 所有触发条件只设置 `should_process = True`，不直接 return
2. 统一在函数末尾判断 `should_process`，为 True 时才执行 ASR 处理
3. `processing = True` 仅在实际处理前设置，确保 `finally` 块能正确重置

**修复后逻辑流程**：
```
收到音频 → 检测到超30秒 → should_process=True → 继续执行
         → 检测到其他条件 → should_process=True/False → 继续执行
         ↓
    if not should_process → return（等待更多数据）
         ↓
    processing=True → 执行ASR处理 → finally: processing=False
```

### 涉及文件

- `main.py` 第314-397行 `on_audio_stream` 函数

## 问题二：音频切片始终为30秒（无法实现2秒切片）

### 现象

- 后端日志始终显示 `Processing audio: 960000 bytes, duration: 30.00s`
- 预期切片时长为2秒，实际始终为30秒
- 识别延迟极高，用户需要等待30秒才能看到结果

### 根本原因

切片判断依赖 `elapsed_since_last_add`（距离上次收到音频包的时间间隔）来检测"语音停顿"，但这个逻辑**从根本上就是错的**：

1. **音频持续流入**：WebSocket 每 ~32ms 发送一个音频包
2. **`last_add_time` 持续更新**：每次收到包都更新为当前时间
3. **`elapsed_since_last_add` 永远接近0**：约32ms，远小于 `END_OF_SPEECH_WAIT`（1.2s）
4. **停顿检测永远不触发**：`END_OF_SPEECH_WAIT` 和 `SILENCE_DURATION_THRESHOLD` 条件永远无法满足
5. **唯一触发条件**：只剩30秒上限 `ASR_MAX_AUDIO_SECONDS`

**原代码逻辑（错误）**：
```python
elapsed_since_last_add = current_time - session.get('last_add_time', 0)
# 音频持续流入 → elapsed 永远 ≈ 0.032s → 永远 < 1.2s → 永远不触发
if elapsed_since_last_add >= END_OF_SPEECH_WAIT:  # 永远为 False
    should_process = True
```

### 解决方案

改为**基于缓冲区时长**直接判断，逻辑清晰且可靠：

| 条件 | 触发时机 | 说明 |
|------|---------|------|
| `audio_duration >= 30s` | 缓冲区超长 | 安全兜底，防止异常 |
| `audio_duration >= 2s` | 缓冲区达到2秒 | **主要触发条件**，实现2秒切片 |
| `audio_duration >= 1s` 且静音 | 短语音+停顿 | 处理1~2秒的短句，说完即识别 |

删除了不再使用的 `SILENCE_DURATION_THRESHOLD`、`END_OF_SPEECH_WAIT` 常量和 `last_add_time` 追踪逻辑。

### 涉及文件

- `main.py` 第222-227行 常量定义
- `main.py` 第337-349行 切片判断逻辑

## 问题三：前端 ScriptProcessorNode 弃用警告

### 现象

浏览器控制台输出：
```
[Deprecation] The ScriptProcessorNode is deprecated. 
Use AudioWorkletNode instead. (https://bit.ly/audio-worklet)
```

### 原因

`ScriptProcessorNode` 是 Web Audio API 中的旧接口，已在现代浏览器中被标记为弃用，推荐使用 `AudioWorkletNode` 替代。旧接口存在以下问题：
- 在主线程上运行，可能造成音频卡顿
- 未来版本可能被移除

### 解决方案

1. 新建 `static/js/audio-processor.js` AudioWorklet 处理器文件
2. `setupAudioProcessing` 改为 `async` 函数，优先使用 `AudioWorkletNode`
3. 若 AudioWorklet 不可用（旧浏览器），自动回退到 `ScriptProcessorNode`

**AudioWorkletNode 优势**：
- 在独立线程运行，不阻塞主线程
- 性能更好，延迟更低
- 是 Web Audio API 的标准推荐方案

### 涉及文件

- `static/js/audio-processor.js`（新建）
- `static/js/app.js` 第226-274行 `setupAudioProcessing` 函数
- `static/js/app.js` 第203行 `startRecording` 函数（添加 await）

## 预期效果

修复后，实时语音识别的流程应为：

```
用户说话 → 麦克风采集 → AudioWorklet处理 → WebSocket发送PCM数据
→ 后端缓冲区累积 → 达到2秒/检测到静音 → 触发ASR API调用
→ 返回识别文字 → 前端实时显示
```

- 切片时长：~2秒（主要）或 ~1秒（短句+静音）
- 识别延迟：2秒内（+ ASR API响应时间）
- 不再出现30秒切片
- 不再出现 ScriptProcessorNode 弃用警告
